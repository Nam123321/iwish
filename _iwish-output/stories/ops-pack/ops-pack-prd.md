## 1. Executive Summary

I-Wish Ops-Pack closes the production lifecycle gap in I-Wish-DragonBall by introducing:
- **Agent Krillin** — a dedicated SRE/DevOps agent persona
- **Android 17** — a dedicated SecOps/Compliance agent persona
- **15+ new workflows** — covering the complete Day 0, Day 1, and Day 2 operational paradigm.
- **1 new library pack** — `templates/library/ops-pack/` with skills and workflows

The module integrates with the existing Whis Capability Management system, enabling automated learning from production incidents.

## 4. User Journeys

### UJ-01: CI/CD Pipeline Setup
**Actor:** Developer  
**Trigger:** Project ready for automated testing  
**Flow:** Call `/setup-ci-pipeline` → Select CI tool (GitHub Actions / GitLab CI) → Select tech stack → Krillin generates CI config file → Developer commits config → CI runs on next push.  
**Success:** Green CI pipeline on first push after setup.

### UJ-02: Production Deployment
**Actor:** Developer or DevOps Engineer  
**Trigger:** Feature branch merged, staging approved  
**Flow:** Call `/deploy-production` → Krillin runs pre-deployment checklist (tests, migrations, env vars, secrets) → Confirm deployment strategy (rolling/blue-green) → Execute deploy → Smoke test → Confirm success.  
**Success:** Zero-downtime deployment completed with audit trail.

### UJ-03: Incident Response  
**Actor:** On-call Engineer  
**Trigger:** Production alert (5xx spike, latency degradation, service down)  
**Flow:** Call `/incident-response` → Krillin structures incident (severity, impact, timeline) → Guide triage → If rollback needed, trigger `/rollback-deployment` → Generate post-mortem → Log instinct to Whis.  
**Success:** Structured incident resolution with post-mortem document and instinct logged.

### UJ-04: Release Management
**Actor:** Team Lead  
**Trigger:** Sprint complete, features ready for release  
**Flow:** Call `/create-release` → Krillin scans commit history → Generates semantic version → Creates changelog → Tags release → Generates release notes from stories.  
**Success:** Tagged release with auto-generated changelog.

### UJ-05: System Maintenance
**Actor:** DevOps Engineer  
**Trigger:** Scheduled monthly maintenance  
**Flow:** Call `/database-maintenance` → Krillin audits DB indexes, suggests vacuum/reindex → Call `/security-scan` → Scan dependencies for CVEs → Report findings.  
**Success:** Maintenance report with actionable items.

## 6. Functional Requirements

### FR-01: DevOps & SecOps Dual-Agents
**Krillin** can be invoked to handle infrastructure, CI/CD, and AIOps. **Android 17** can be invoked to handle security scanning, secret management, and compliance enforcement. Both follow I-Wish DSL conventions and strict guarded routing.

### FR-02: CI Pipeline Setup
Given a target CI/CD tool (GitHub Actions or GitLab CI) and tech stack (Node.js, Python, Go, etc.), Krillin generates a valid CI configuration file with stages: lint, test, build. The generated file passes the CI tool's own schema validation.

### FR-03: Pre-Deployment Checklist
Before any deployment, Krillin runs a 10-point checklist covering: test suite pass, migration status, environment variables, secrets rotation, API compatibility, performance baseline, rollback plan, monitoring readiness, backup verification, team notification.

### FR-04: Staging Deployment
Krillin generates a deployment script or configuration for the target platform (Docker Compose, SSH, Kubernetes, Vercel CLI). The workflow includes health check verification after deployment.

### FR-05: Production Deployment & Veto Protocol
Krillin supports rolling update and blue/green deployment. **HOTL Integration:** Before initiating prod deployment, Krillin MUST generate a "Decision Summary" (scope, risk, rollback plan) and explicitly pause for Human Veto (User Approval). Zero-downtime is required.

### FR-06: Monitoring Setup
Krillin generates monitoring configuration for at least 3 signal types (Metrics, Logs, Traces). Platform-adaptive: supports Prometheus/Grafana, Datadog.

### FR-07: Incident Response Protocol (AIOps & RCA)
Krillin structures incident handling into 5 phases: Detection → Triage → Mitigation → Resolution → Post-Mortem. **AIOps:** Krillin automatically groups correlated alerts to identify the Root Cause. Post-mortem follows blameless format (5 Whys). Instinct is automatically logged to `.agent/memory/instincts.jsonl`.

### FR-08: Rollback Deployment
Krillin can rollback application code within 5 minutes. Includes telemetry monitoring to suggest auto-rollback if error rates spike immediately post-deployment.

### FR-09: Release Management
Krillin generates semantic version based on commit types. Auto-generates changelogs.

### FR-10: FinOps & Self-Healing (Vision)
Krillin analyzes cloud resource utilization to suggest rightsizing (FinOps). Provides runbook automation to restart pods or clear caches upon specific trigger (Self-Healing).

### FR-10: Database Maintenance
Krillin generates maintenance scripts for: index analysis and optimization, table vacuum/analyze, connection pool audit, slow query identification. Platform-adaptive: supports PostgreSQL, MySQL, MongoDB.

### FR-11: Security Scanning
Krillin runs `npm audit` / `pip audit` / `go mod audit` based on detected tech stack. CVE results are formatted with severity, affected package, and remediation steps. Critical CVEs are flagged for immediate action.

## 7. Non-Functional Requirements

### NFR-01: Platform Adaptability
Ops-Pack workflows must support at minimum 3 deployment targets: VPS (SSH/Docker), Container Orchestration (Kubernetes/Docker Swarm), PaaS (Vercel/Railway/Render). Workflows detect the target from project configuration.

### NFR-02: Token Efficiency
Each workflow file must be under 500 lines. Agent persona under 100 lines. All machine-generated outputs (instincts, reports) use Dense JSONL format per I-Wish Memory Schema.

### NFR-03: Human-on-the-Loop (HOTL) Guardrails
Krillin operations are bound by strict Guardrails:
1. **Least Privilege:** Agent operates with task-scoped credentials, never standing God-mode access.
2. **Step / Cost Thresholds:** Infinite loop prevention via strict tool call limit bounding per workflow.
3. **Immutable Audit Trails:** All production-modifying actions executed by Krillin must be logged definitively within the issue tracker or system logs before execution.

### NFR-04: Idempotency
Deployment workflows must be idempotent — running the same deployment twice produces the same result without side effects.

### NFR-05: Whis Integration
Every workflow that detects an error, anomaly, or lesson learned must append a Dense JSONL instinct to `instincts.jsonl`.
