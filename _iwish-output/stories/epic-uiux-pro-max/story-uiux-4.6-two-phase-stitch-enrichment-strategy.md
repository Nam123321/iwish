# Story UIUX-4.6: Two-Phase Stitch Enrichment Strategy

## 1. Context & Objective
As part of the UI/UX Pro Max Specialist integration, we have discovered a gap between the static HTML/CSS generated by Stitch and the dynamic, highly interactive capabilities defined in our comprehensive Pro Max library. To solve this, we are implementing a **Two-Phase Stitch Enrichment Strategy**. 

The objective is to decouple the initial visual layout generation (Phase 1) from the enrichment of micro-interactions and logic (Phase 2), utilizing a Prompt Injection pattern. This allows the UI Spec to serve as a single source of truth for both static and dynamic requirements.

## 2. Acceptance Criteria (AC)
- **AC1 (Template Update):** The UI Spec template (`template.md`) must be updated to include two new sections: `[STITCH_PROMPT_INJECTION]` and `[POST_STITCH_ENRICHMENT_LOGIC]`.
- **AC2 (Workflow Patching):** The `workflow-entry.md` and `iwish-bmm-code-review.md` must enforce the inclusion of these sections and validate their implementation during code review.
- **AC3 (Enrichment Workflow):** A new workflow, `iwish-bmm-enrich-ux.md`, must be created to handle the Phase 2 enrichment logic.
- **AC4 (Skill Updates):** The `ui-ux` Skill must be updated to output the required `[STITCH_PROMPT_INJECTION]` block based on its interaction library.
- **AC5 (Validation):** The implementation must be successfully tested on a sample (e.g., the Distro Landing Page) to confirm both phases execute flawlessly.
- **AC6 (Auto-Trigger Touchpoints):** Master Design defaults to mandatory enrichment. UI Spec generation uses a 3-level scoring system to determine if enrichment is required (Enrich-UX Score).
- **AC7 (Dev Readiness Gate):** Vegeta Dev workflow (`iwish-bmm-dev-story.md`) halts if UI Spec has `Enrichment_Required: true` but `[POST_STITCH_ENRICHMENT_LOGIC]` is empty, unless user types `APPROVE_OVERRIDE`.
- **AC8 (Orchestrator Role):** The `ui-ux` agent acts as the Domain-Specific Orchestrator for Frontend/Design, managing the UI Spec -> Stitch -> Enrich-UX execution pipeline to reduce user cognitive load.
- **AC9 (Visual Fidelity Gate):** Refactor `stitch-design-taste` to `visual-fidelity-gate` and broaden the "Absolute Source of Truth" to be the "Approved Master Design Source" (e.g. Figma, Stitch, Custom UI Kit) rather than hardcoding Stitch.

## 5. Dev Notes (Project Memory)
- **Tracer Bullet:** The vertical slice for this story involves generating a UI spec with the prompt injection, passing it through Stitch, and then enriching the approved UI spec with dynamic JavaScript/CSS logic, which is then strictly verified by the code review agent.
- Ensure the prompt injection block focuses only on instructions Stitch can parse effectively without breaking the static layout generation.

## 6. Enrich-UX Scoring Criteria
To determine if a UI Spec requires enrichment, the `ui-ux` agent will calculate the following score (0-10):

| Factor | Description (Score 0-5) | Weight |
|---|---|---|
| **E1: Interaction Complexity** | 0=Static CRUD, 2=Hovers/State transitions, 4=Drag-drop/Data viz, 5=High-interaction SaaS Landing (GSAP/Scrolljacking) | x2 |
| **E2: Emotional Impact (WOW Factor)** | 0=Admin settings, 2=Efficiency Dashboard, 4=Webstore checkout, 5=Acquisition/Conversion funnels | x2 |
| **E3: Feedback Density** | 0=Standard buttons, 3=Complex counters/Progress, 5=Continuous AI feedback loops (Pulse/Glow) | x1 |
| **E4: Brand Kinetic Identity** | 0=Standard Tailwind/AntD, 2=Custom token tweaks, 5=Core brand kinetic signatures (e.g. Distro Neon Mint glow) | x1 |

**Formula:** `Score = (E1*2 + E2*2 + E3 + E4) / 30 * 10`

**Decision Matrix:**
- **0-3: SKIP ENRICHMENT** (Static HTML/CSS is sufficient).
- **4-6: STANDARD ENRICHMENT** (Inject CSS transitions, hover states, fade-ins).
- **7-10: PRO-MAX ENRICHMENT** (Inject JS Observers, GSAP/Framer, complex state choreography).
