2. Why it exists

Problem: Standard DOM-based text measurement (e.g. getBoundingClientRect()) triggers expensive browser layout reflows, causing substantial scroll jank and framerate drops when rendering dynamic lists.

Solution: Pretext implements a custom multiline text layout and measurement library in pure JS/TS. It shapes segments off-screen using a single Canvas context, cache-stores the metrics, and wraps lines using arithmetic calculations without ever touching the DOM.

3. Delivery Framework Placement

  • Phases Helped: implement, validate
  • Stages Served: UI & Layout implementation, Visual validation
  • Role: supportive client utility library.

4. Use Cases

Category Description
Core Virtualized scroll lists sizing, Masonry layouts, Obstacle-aware text routing (flowing text around shapes).
Adjacent Offline visual checks: validating whether text buttons or badges overflow/wrap without running a browser session.
Exclusions Full WYSIWYG markup rendering, Node.js backend rendering lacking a canvas implementation, system-ui font stacks.

5. Constraints & Edge Cases

macOS Canvas Emoji discrepancy: Chrome and Firefox canvas measures emoji wider than DOM at font sizes <24px on macOS. Pretext addresses this by executing a one-time DOM calibration read, creating an intermittent DOM dependency.

System Fonts: The system-ui macOS font stack resolves differently in Canvas vs. DOM. Use named fonts for guaranteed accuracy.

Intl.Segmenter: The engine requires browser support for Intl.Segmenter. Runtimes without it must be polyfilled.

6. Example usage

import { prepare, layout } from '@chenglou/pretext'

// Phase 1: Prepare (Heavy calculation run once)
const prepared = prepare('AGI 春天到了. بدأت الرحلة 🚀', '16px Inter')

// Phase 2: Layout (Super fast arithmetic on resize)
const { height, lineCount } = layout(prepared, 320, 20)

7. User Review Questions

  1. Are you planning to run layout measurements in backend environments (e.g. Node serverless)? If yes, ensure Canvas polyfills are registered.
  2. Does your font stack use macOS system font stacks? If yes, consider switching to named webfonts to prevent layout variance.
