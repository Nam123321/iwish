Integration Guide: strix

Adoption Review Pack & Context Reference
Shape: Skill Pack Role: Security Validation State: Absorbed

🛡️ 1. Overview & Purpose

Strix is an open-source, autonomous application security validation agent. Unlike standard vulnerability scanners that generate high-noise static checklists, Strix acts like an AI-hacker: discovering security flaws, attempting to construct safe Proofs-of-Concept (PoCs) to validate exploitability, and generating remediation code.

By absorbing Strix into the I-Wish Platform, we introduce native application-security checks into codebases directly during development and testing workflows.

⚙️ 2. Delivery Framework Placement

Strix is integrated as a supportive/auxiliary skill pack that works alongside existing agents. It operates in the following stages:

Phase Active Stage Role
Validate Triaging pull request security alerts & writing verification unit tests. process-primary (for security audits)
Solution Advising on configuration hardening and threat modeling. supportive
Operate & Learn Executing safe active probing in sandboxes. supportive

🔄 3. Execution Pipeline

📥 Expected Inputs
  • target_url: Live environment for active verification probes.
  • local_code_path: Root directory for code audits.
  • vulnerability_type: Targeted OWASP vulnerability.
📤 Expected Outputs
  • vulnerability_report: Structured details with CVSS metrics.
  • poc_payload: Replicable curl payload or script.
  • remediation_patch: Code diff containing fix.

💡 4. Target Use Cases

Core Use Cases: Automated verification of pull requests, active vulnerability testing against sandboxed test suites, and dynamic validation to eliminate false positives from static code analysis (SAST).
Adjacent Use Cases: Auditing configuration parameters (e.g. Firebase Firestore rules, NestJS CORS config) and inspecting Docker/Kubernetes files for local privilege escalations.
Do-Not-Use Cases: Never use for general coding assistance, non-security features, or executing scanners against domains/URLs not registered inside the platform's verified target scope.

⚠️ 5. Stress Cases & Operational Constraints

  • WAF Evasion: Targets behind active WAFs may block scanner probes, leading to false negatives. Requires using advanced bypass techniques.
  • DoS Safety: High concurrency active fuzzer checks (e.g. ffuf sprays) can overwhelm small dev servers. Concurrency limits are hardcoded.
  • Sandbox Isolation: All active validation scripts MUST execute strictly inside isolated containers to prevent unintended damage to the host machine.

📖 6. Example Scenarios

Scenario 1: Pre-merge API Audit

User Prompt: "Verify if our authentication routing in NestJS is secure before merging this PR."

Orch Action: Invokes review-agent, loads strix/skills/frameworks/nestjs.md, scans routing parameters, and runs local unit tests to verify JWT validations.

Scenario 2: IDOR Validation

User Prompt: "A SQL issue was reported in our user retrieval function. Audit the code."

Orch Action: Loads strix/skills/vulnerabilities/sql_injection.md, runs Semgrep to isolate database queries, and verifies if parameters are properly escaped.
