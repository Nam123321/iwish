# Terminal Safety — Agent Command Execution Rules

> **Trigger:** This fragment is loaded by Vegeta and Songoku agents before executing any terminal commands during story implementation. It is also referenced by the Iron Law of Debugging fragment.

---

## Rule 1: Loop Guard

Every `while` or `for` loop generated by an Agent in a bash script MUST include a safety counter:

```bash
# ✅ CORRECT — bounded loop
MAX_ITERATIONS=1000
counter=0
while [ some_condition ]; do
  counter=$((counter + 1))
  if [ $counter -ge $MAX_ITERATIONS ]; then
    echo "ERROR: Loop exceeded $MAX_ITERATIONS iterations. Terminating."
    exit 1
  fi
  # ... loop body ...
done
```

```bash
# ❌ FORBIDDEN — unbounded loop
while true; do
  # ... no exit condition ...
done
```

**Exception:** `while read -r line` loops reading from a finite file are exempt (the file itself is the bound).

## Rule 2: Command Timeout

When using `run_command` tool:

- **Short commands** (ls, grep, find, cat, wc): `WaitMsBeforeAsync` = 2000–5000ms.
- **Build commands** (npm run build, tsc, pnpm build): `WaitMsBeforeAsync` = 10000ms (max).
- **Dev servers** (npm run dev, pnpm dev): MUST use `RunPersistent: true` and `WaitMsBeforeAsync` = 2000ms. Never wait synchronously for a dev server.

If a command is expected to take > 10 seconds:
1. Send it to background (`WaitMsBeforeAsync` = 2000).
2. Poll with `command_status` at intervals.
3. If no output after 3 polls (90 seconds total) → **terminate** the command and investigate.

---

## Rule 3: No-Watch Mode

When running build tools or linters:

- ✅ **Prefer** single-run mode: `tsc --noEmit`, `eslint .`, `jest --watchAll=false`
- ❌ **Avoid** watch mode: `tsc --watch`, `jest --watch`, `nodemon`
- If watch mode is absolutely required (e.g., dev server), it MUST be run with `RunPersistent: true`.

**Rationale:** Watch-mode commands never exit, causing the Agent to hang indefinitely waiting for completion.

---

## Rule 4: Output Kill Switch

When reading command output via `command_status`:

- Set `OutputCharacterCount` to the **minimum needed** (typically 2000–5000 characters).
- If a command produces > 50,000 characters of output and is still running:
  1. **Terminate** the command immediately.
  2. Log: `"Command terminated: excessive output (>50k chars). Likely infinite loop or verbose debug mode."`
  3. Re-run with output filtering (e.g., `| tail -50` or `| head -100`).

---

## Rule 5: Destructive Command Awareness

Before executing any command that matches these patterns, Agent MUST pause and confirm intent:

| Pattern | Risk | Required Action |
|---|---|---|
| `rm -rf` | File deletion | Confirm path is scoped, never use on `/` or `~` |
| `DROP TABLE`, `DROP DATABASE` | Data destruction | Require explicit user approval |
| `git push --force` | History rewrite | Require explicit user approval |
| `chmod -R 777` | Security degradation | Reject — suggest specific permissions |
| `kill -9` | Process termination | Confirm PID is correct |

This rule supplements (not replaces) the Iron Law of Debugging's Scope Lock principle.
