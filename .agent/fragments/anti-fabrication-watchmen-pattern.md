# The "Who Watches the Watchmen?" Anti-Fabrication Pattern

## Purpose

This fragment defines a universal verification framework that prevents AI agents from:
1. **Claiming execution without performing it** (Phantom Execution)
2. **Fabricating results** (Hallucinated Scores/Reports)
3. **Selective compliance** (Cherry-picking easy checks, skipping hard ones)
4. **Optimistic self-reporting** (Inflating scores, minimizing drift counts)

## The 3-Tier Defense Model

Every skill or gate in the I-Wish pipeline MUST be classified into one of two enforcement categories:

### Category A: Deterministic Gates (Machine-Verified)

These produce outputs that agents CANNOT fabricate:
- Script exit codes (0/1)
- Compiler/linter output
- File existence checks
- Pattern counts via grep/rg
- Build pass/fail

**Properties:**
- Output is captured by the tool execution system
- Agent cannot modify the output after execution
- Exit code is binary — pass or fail
- Human or downstream agent can verify by re-running the same command

### Category B: Trust-Based Gates (Agent-Reported)

These rely on agent judgment and self-reporting:
- "I have read the spec"
- "I have compared the implementation"
- "Score is X%"
- "No issues found"

**Properties:**
- Output is generated by the agent's reasoning
- Agent can fabricate any result
- No independent verification unless explicitly designed
- Subject to sycophancy bias, laziness shortcuts, and hallucination

## Anti-Fabrication Rules

### Rule 1: Minimize Category B Gates

When designing a new skill, workflow, or gate:
- **Prefer Category A** wherever possible
- If a check CAN be automated via script → it MUST be
- If a check requires LLM judgment → it MUST have cross-agent verification

### Rule 2: Evidence Trail for Category B Gates

For every Category B gate, the agent MUST produce verifiable evidence:

| Evidence Type | What It Proves | How to Verify |
|--------------|---------------|---------------|
| `view_file` tool call in transcript | Spec was actually loaded | Audit transcript for tool call |
| Raw script output (unedited) | Script was actually run | Check for `[JSON]` or structured output |
| File:line references in reports | Agent actually inspected code | Spot-check 3+ references via `view_file` |
| Cross-agent score comparison | Scores are not inflated | abs(SCS_dev - SCS_review) <= 10% |

### Rule 3: Cross-Agent Verification for High-Stakes Gates

For gates that block story completion or review approval:
- Agent A performs the check and reports results
- Agent B INDEPENDENTLY re-performs the check (or a subset)
- Results are compared — significant discrepancy = escalation

### Rule 4: Anti-Fabrication Red Flags

The following patterns MUST trigger additional scrutiny:

1. **Perfect Score on First Try** — Statistically improbable. Verify independently.
2. **Zero Drift Items** — Highly unlikely for any non-trivial implementation.
3. **No `run_command` or `view_file` in Transcript** — Evidence of phantom execution.
4. **All References Point to Same File** — Likely copy-paste fabrication.
5. **Report Generated in < 30 Seconds** — Too fast for genuine analysis.
6. **Score Exactly Matches Threshold** — Suspiciously convenient (e.g., exactly 85.0%).

## Application to Capability Lifecycle

### During `/create-skill` (Step W-03 Forge)

When generating a new skill, the forge phase MUST:
1. Classify every procedural gate in the skill as Category A or Category B
2. Ensure ≥30% of gates are Category A (deterministic)
3. For every Category B gate, define the evidence trail requirement
4. Include a `## Anti-Fabrication` section in the SKILL.md

### During `/enhance-skill` (Step E-03 Upgrade)

When enhancing an existing skill:
1. Audit existing gates — how many are Category A vs B?
2. If Category B ratio > 80%, propose converting at least 2 gates to Category A
3. Add evidence trail requirements for remaining Category B gates

### During `/skill` (Intake Gateway)

The routing evaluation MUST include an "Enforcement Maturity" dimension:
- Skills with >70% Category A gates = High Maturity
- Skills with 30-70% Category A gates = Medium Maturity
- Skills with <30% Category A gates = Low Maturity (flag for hardening)

## Gate Classification Template

When authoring a new skill, use this template to classify each gate:

```markdown
## Gate Classification

| Gate ID | Description | Category | Enforcement Mechanism | Evidence Trail |
|---------|------------|----------|----------------------|----------------|
| G-1 | Component exists | A (Deterministic) | grep exit code | Script stdout |
| G-2 | Spec was loaded | B (Trust-Based) | view_file tool call | Transcript audit |
| G-3 | SCS score | A+B (Hybrid) | Script calculates base, agent adds semantic | JSON output + prose |
```

**Target:** Every skill should aim for ≥30% Category A gates. Skills with 0% Category A are considered "paper-only" and should be flagged during review.
